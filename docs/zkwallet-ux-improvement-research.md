# zkWallet 초보자 친화적 UI/UX 개선 연구

> 작성일: 2026-03-17
> 목적: ZK 비밀 송금의 진입 장벽을 낮추고, Venmo/Cash App 수준의 사용 경험을 제공하기 위한 UX 연구

---

## 1. 현재 상태 분석

### 1.1 현재 송금 플로우 (5단계)

```
TokenTransfer 화면
  ├── 1. 토큰 선택 (잔액 확인: 공개 + 비공개)
  ├── 2. 입력 수집
  │     ├── Public amount (공개 지갑에서)
  │     ├── Private amount (ENA/비공개 계정에서)
  │     ├── Private note 선택 (이전 전송 출력 노트 연결)
  │     ├── 수신자 주소/지갑 선택
  │     ├── Public send amount (수신자가 공개적으로 받는 금액)
  │     └── Private send amount (비공개 노트로 보내는 금액)
  ├── 3. 확인 모달 (TransferConfirmModal)
  ├── 4. 진행 추적 (TransferActivityModal: 4-5단계)
  │     ├── Init Transfer
  │     ├── Generate Proof
  │     ├── Send to Network
  │     ├── Mining
  │     └── Fetch Balance
  └── 5. 완료 화면 (TransferComplete: 폭죽 애니메이션)
```

### 1.2 현재 UX의 문제점

| 문제 | 심각도 | 설명 |
|------|--------|------|
| **입력 필드 과다** | 높음 | Public/Private 입력 + 출력을 사용자가 직접 분배해야 함 |
| **개념 노출** | 높음 | "Public amount", "Private amount", "Note" 등 ZK 개념이 화면에 노출 |
| **수신자 주소 직접 입력** | 높음 | 긴 hex 주소를 복사/붙여넣기해야 하는 경우 발생 |
| **잔액 이중 표시** | 중간 | 공개/비공개 잔액을 사용자가 구분해서 이해해야 함 |
| **증명 생성 대기** | 중간 | "Generate Proof" 단계에서 기술적 용어 노출 |
| **네트워크 선택 필요** | 중간 | 초보자에게 불필요한 결정 요구 |
| **가스비 별도 관리** | 낮음 | ETH/KAIA를 별도로 보유해야 가스비 지불 가능 |

### 1.3 기존 UX 원칙 (이미 정의됨, 일부 미구현)

기존 `zktransfer-consumer-ux.md`에서 정의한 원칙:
- 사용자는 **누구에게, 무엇을, 얼마를** 만 결정
- ZK/proof/note/witness/circuit 용어 숨김
- 자동 추천 모드 제공

**문제**: 이 원칙이 현재 모바일 앱에 완전히 반영되지 않았음.

## 2. 벤치마크: 선행 사례 분석

### 2.1 Zcash (Zashi Wallet) — "Default Private"

- **Unified Address**: 사용자에게 하나의 주소만 노출. 내부적으로 최신 shielded pool로 자동 라우팅
- **결과**: Shielded ZEC 비율이 11% → ~30%로 증가 (2025년)
- **교훈**: **프라이버시를 기본값으로 설정하면 별도 교육 없이 채택률이 3배 증가**

### 2.2 Railgun (Railway Wallet) — "Shield/Unshield"

- Shield(차폐) → 사용 → Unshield(해제) 의 명시적 흐름
- 수수료 명확 표시 (0.25% + gas)
- Broadcaster를 통한 전송으로 익명성 보장
- **교훈**: Shield/Unshield 패러다임은 추가 마찰을 만듦. **차폐를 기본 상태로 만드는 것이 더 나음**

### 2.3 Venmo / Cash App — "금융 앱의 황금 표준"

- **3탭 전송**: 사람 선택 → 금액 입력 → Pay 터치
- 연락처 기반 주소 지정 (hex 주소 불필요)
- 법정화폐 기준 표시 (토큰 단위는 부수적)
- 엄지 존(thumb zone) 하단에 핵심 액션 배치
- **교훈**: 이것이 목표 UX. 모든 기술적 복잡성은 이 수준 아래에 숨겨야 함

### 2.4 Aztec Network — "Client-Side Proving"

- 모든 증명을 클라이언트 디바이스에서 생성 (데이터가 사용자 폰을 떠나지 않음)
- 수수료 추상화: ETH 또는 USDC로 수수료 지불 가능
- **교훈**: 클라이언트사이드 증명은 프라이버시에 유리하나 모바일 성능 도전. 명확한 진행 표시 필수

## 3. 제안: 개선된 UI/UX 설계

### 3.1 핵심 원칙

```
┌─────────────────────────────────────────────────────────┐
│                    5대 설계 원칙                          │
├─────────────────────────────────────────────────────────┤
│ 1. 프라이버시는 기본값이다 (옵트인 아님)                    │
│ 2. 사용자는 '무엇'만 결정, 시스템이 '어떻게' 결정            │
│ 3. 기술 용어 제로 — 일상 언어만 사용                        │
│ 4. 3탭 이내에 송금 완료                                    │
│ 5. 신뢰는 투명성으로 구축 (교육이 아님)                      │
└─────────────────────────────────────────────────────────┘
```

### 3.2 개선된 송금 플로우 (Before → After)

#### Before (현재: 5단계, 6+ 입력 필드)

```
[토큰 선택] → [Public 금액 입력] → [Private 금액 입력]
→ [Note 선택] → [수신자 주소 입력] → [Public/Private 출력 분배]
→ [확인] → [5단계 진행 모달] → [완료]
```

#### After (제안: 3탭, 2 입력 필드)

```
[송금 버튼] → [수신자 + 금액 입력] → [Face ID 확인] → [완료]
```

### 3.5 용어 치환 가이드 (전체)

| 현재 (기술 용어) | 제안 (일상 언어) | 사용 위치 |
|-----------------|-----------------|----------|
| Public Amount | 보내는 금액 | 송금 화면 |
| Private Amount | (표시 안 함 — 시스템 자동) | 내부 처리 |
| ENA (Encrypted Note Account) | (표시 안 함) | 내부 처리 |
| Private Account | 내 잔액 | 홈 화면 |
| Public Account | (표시 안 함 — 통합 잔액) | 홈 화면 |
| Generate Proof | 보안 검증 중 | 진행 화면 |
| Mining | 전송 완료 대기 중 | 진행 화면 |
| Note Selection | (자동 선택 — UI 없음) | 내부 처리 |

## 4. Progressive Disclosure (점진적 노출) 설계

### Tier 1 — 초보자 (기본값)

사용자에게 보이는 것:
- 통합 잔액 (법정화폐)
- 송금/받기 버튼
- 거래 내역 (일상 언어)
- "🔒 보호됨" 상태 배지

사용자에게 숨기는 것:
- 공개/비공개 잔액 분리
- 네트워크 이름
- 토큰 컨트랙트 주소
- 가스 파라미터
- 증명 상세 정보

### Tier 2 — 중급 (탭해서 확장)

거래를 탭하면 추가로 보이는 것:
- 확인(confirmation) 횟수
- 증명 상태 (성공/실패)
- 사용된 네트워크
- 수수료 상세 분류 (보안 수수료 + 네트워크 수수료)

설정에서 접근 가능:
- 토큰 관리
- 네트워크 전환
- 연락처 관리

### Tier 3 — 고급 (설정에서 활성화)

"개발자 모드" 또는 "고급 모드" 토글:
- Raw 트랜잭션 데이터
- 증명 상세 (proof artifacts)
- 수동 가스 조정
- 커스텀 RPC 엔드포인트
- Public/Private 금액 수동 분배
- Note 수동 선택
- 증명 내보내기

### 5.1 자동 송금 방식 결정 로직

현재는 사용자가 Public/Private 금액을 직접 분배해야 합니다. 이를 자동화하는 알고리즘:

```typescript
// 제안: 자동 송금 방식 결정
function determineTransferMode(
  amount: BigInt,
  publicBalance: BigInt,
  privateBalance: BigInt,
  receiverHasENA: boolean
): TransferMode {

  // Case 1: 수신자가 ENA를 가지고 있고, 비공개 잔액이 충분하면 → 비공개 송금
  if (receiverHasENA && privateBalance >= amount) {
    return {
      mode: 'private',
      inputPrivate: amount,
      inputPublic: 0n,
      outputPrivate: amount,
      outputPublic: 0n,
      label: '🔒 비공개 송금',
    };
  }

  // Case 2: 비공개 잔액 부족하면 혼합 (비공개 우선)
  if (receiverHasENA && privateBalance > 0n) {
    const fromPrivate = privateBalance;
    const fromPublic = amount - privateBalance;
    return {
      mode: 'mixed',
      inputPrivate: fromPrivate,
      inputPublic: fromPublic,
      outputPrivate: amount,
      outputPublic: 0n,
      label: '🔒 비공개 송금 (혼합)',
    };
  }

  // Case 3: 비공개 잔액 없거나 수신자 ENA 없음 → 공개 송금
  return {
    mode: 'public',
    inputPrivate: 0n,
    inputPublic: amount,
    outputPrivate: 0n,
    outputPublic: amount,
    label: '⚡ 빠른 송금',
  };
}
```

### 5.2 Note 자동 선택 로직

```typescript
// 제안: 자동 Note 선택 (FIFO + 최적 조합)
function autoSelectNotes(
  targetAmount: BigInt,
  availableNotes: Note[]
): SelectedNotes {
  // 금액 기준 정렬 (큰 것 우선 — 잔돈 최소화)
  const sorted = availableNotes.sort((a, b) =>
    Number(b.value - a.value)
  );

  let remaining = targetAmount;
  const selected: Note[] = [];

  for (const note of sorted) {
    if (remaining <= 0n) break;
    selected.push(note);
    remaining -= note.value;
  }

  return {
    notes: selected,
    totalValue: targetAmount - remaining,
    change: remaining < 0n ? -remaining : 0n,
  };
}
```

## 10. Seed Phrase 없는 지갑: 보안 분석 및 하이브리드 전략

### 10.1 문제 제기: Seed Phrase는 정말 필요한가?

ZK 비밀 송금의 진입 장벽을 낮추려면, 가장 큰 UX 장벽인 seed phrase(복구 문구)를 어떻게 처리할지 결정해야 합니다. 이 섹션에서는 seed phrase 유지/제거의 실질적 위험을 분석하고 zkWallet에 최적인 하이브리드 접근을 제안합니다.

### 10.2 Seed Phrase 유지의 현실적 위험

| 위험 | 데이터 |
|------|--------|
| 영구 분실 | 전체 비트코인의 ~20%가 영구 분실 (약 400만 BTC, $93B 이상) |
| 사용자 이해도 부족 | 사용자 중 43%만 seed phrase의 의미를 정확히 이해 (2025 CHI 학술 연구) |
| 피싱/사회공학 | 2025년 보안 사고의 40.8%가 사회공학 공격 (주로 seed/키 탈취) |
| 지갑 탈취 규모 | 2025년 상반기에만 $17.1억 피해 (69%가 키/seed 도난) |
| AI 피싱 확산 | AI 기반 딥페이크 음성/자동 스캠 사이트가 급증, 2026년 1월에만 $3억 피싱 피해 |

**핵심**: Seed phrase는 이론적으로 완벽한 자기 주권 도구이지만, 대다수의 일반 사용자는 "자기 은행의 보안팀" 역할을 수행할 준비가 되어 있지 않습니다.

### 10.3 Seed Phrase 제거의 현실적 위험

| 위험 | 설명 |
|------|------|
| **MPC 제공자 의존** | 키 share를 보유한 서버가 다운되면 접근 불가. 2023년 BitForge 취약점이 15+ 지갑 제공자에 영향 |
| **Passkey 플랫폼 종속** | Apple/Google 계정이 탈취되면 패스키도 함께 탈취됨. 클라우드 동기화의 보안은 해당 플랫폼에 의존 |
| **벤더 락인** | MPC 키를 다른 제공자로 이전 시 전체 키가 한 지점에서 재조합 → MPC 보안 가정이 붕괴됨 |
| **규제 불확실성** | 키 share를 보유한 제공자가 수탁자(custodian)로 분류될 수 있어 자산 동결/공개 명령 위험 |
| **성숙도 격차** | Seed phrase 15년+ vs MPC ~5년, Passkey ~2년, Smart Account ~3년 — 아직 검증 기간이 짧음 |
| **제공자 폐업 위험** | MPC 제공자가 사전 내보내기 도구 없이 폐업하면 자산 영구 접근 불가 |

### 10.4 대안 기술 비교

```
├── 클라우드 암호화 백업 (iCloud Keychain / Google Drive)
├── 72시간 시간 지연으로 악의적 복구 시도 방지
└── 알림: 복구 시도 시 모든 기기에 즉시 알림 발송

계층 3 — 자기 주권 (고급 사용자, 설정에서 선택적 활성화)
├── 설정 > 고급 > "복구 문구 내보내기"
├── 경고 표시: "이 문구를 분실하거나 노출하면 자산을 잃을 수 있습니다"
├── Shamir's Secret Sharing (3-of-5 분할) 옵션
│   └── 5개 조각 중 3개만 있으면 키 복원 가능
└── 전통적 12/24 단어 seed phrase 내보내기 지원
```

#### 이 접근이 안전한 이유

1. **초보자는 seed phrase를 다룰 필요 없음** → 분실/피싱 위험 제거 (연간 수십억 달러 손실 방지)
2. **소셜 리커버리가 보험** → 디바이스 분실 시에도 신뢰 연락처를 통해 복구 가능
3. **고급 사용자는 자기 주권 유지** → seed 내보내기가 여전히 가능하여 자율성 보존
4. **다중 방어선** → MPC + 소셜 리커버리 + (선택적) seed — 어느 한 경로가 실패해도 다른 경로 존재
5. **시간 지연 + 알림** → 악의적 복구 시도를 사전에 감지하고 차단할 수 있음

#### 디바이스 분실 시나리오별 복구 경로

| 시나리오 | 계층 1 복구 | 계층 2 복구 | 계층 3 복구 |
|---------|------------|------------|------------|
| 폰 분실 (Apple ID 유지) | ✅ 새 기기에서 소셜 로그인 + MPC share 복원 | ✅ 클라우드 백업 | ✅ Seed phrase |
| 폰 분실 + Apple ID 분실 | ❌ 불가 | ✅ 소셜 리커버리 (2/3 연락처 승인) | ✅ Seed phrase |
| 폰 분실 + Apple ID 분실 + 연락처 비협조 | ❌ | ❌ | ✅ Seed phrase/Shamir 조각 |
| 모든 백업 분실 | ❌ | ❌ | ❌ (이 경우만 자산 접근 불가) |

**결론**: 3계층 모두 실패해야만 자산에 접근할 수 없으며, 이는 seed phrase만 사용하는 기존 지갑보다 훨씬 견고한 안전망입니다.

### 10.7 구현 시 주의사항

1. **MPC 제공자 선택**: BitForge 취약점 패치가 완료된 검증된 제공자 사용 (ZenGo, Fireblocks 등)
2. **키 내보내기 보장**: 제공자 폐업 시에도 키를 복구할 수 있는 탈출 경로(escape hatch) 반드시 구현
3. **소셜 리커버리 가디언 관리**: 가디언 변경 시 시간 지연 적용 (즉시 변경 불가)
4. **클라우드 백업 암호화**: 사용자의 생체 인증 키로 암호화 후 저장 (플랫폼 제공자도 복호화 불가)
5. **정기 보안 감사**: MPC 프로토콜 구현에 대한 외부 보안 감사 필수 (연 1회 이상)
6. **규제 대응**: 키 share 보유에 대한 관할권별 규제 변화 모니터링
